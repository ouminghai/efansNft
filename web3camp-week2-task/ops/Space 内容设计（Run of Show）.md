# 活动当天内容流程：用 AI 辅助完成 Builder 学习记录

## 活动标题

用 AI 辅助完成 Builder 学习记录：从零散学习到可展示 Proof of Work

## 活动简介

这是一场面向 Web3 / AI / Builder 新人的小型线上活动，适合 30-100 人参与。

活动重点不是教大家完整开发一个项目，而是帮助新人理解：如何把一次学习、一次提问、一个小 Demo 或一次项目协作，整理成可以展示的学习记录和 Proof of Work。

很多新人已经开始使用 AI 工具，但常见问题是只停留在问答层面，没有把 AI 的帮助转化成自己的输出。这场活动会围绕“学习记录怎么写、项目文档怎么整理、角色能力怎么表达”展开，让参与者知道如何从零散学习进入 Builder 状态。

## 时间安排

活动总时长建议控制在 60 分钟。

| 时间 | 环节 | 内容重点 |
|---|---|---|
| 0-8 分钟 | 暖场案例 | 用一个新人故事抛出悬念：为什么学了很多却没有 Proof of Work |
| 8-15 分钟 | 问题引入 | 新人为什么学了很多但没有可展示成果 |
| 15-30 分钟 | 案例拆解 | 展示如何用 AI 整理学习笔记、开发计划和角色说明 |
| 30-45 分钟 | 核心讨论 | 围绕 3-5 个问题展开嘉宾讨论或主持提问 |
| 45-55 分钟 | 互动与 Q&A | 参与者现场写一个 3 句话 Proof of Work 草稿，并回到暖场案例做分析 |
| 55-60 分钟 | 收尾 CTA | 鼓励活动后完成一份公开学习记录或 GitHub 文档 |

## 主持串场

### 暖场案例 0-8 分钟

大家好，欢迎来到今天的线上分享。今天的主题是“用 AI 辅助完成 Builder 学习记录”。这场活动不是一场技术课，也不是要大家立刻做出完整产品，而是想解决一个更基础的问题：新人如何把自己正在学、正在做、正在思考的东西，整理成别人能看懂的 Proof of Work。

今天比较适合刚进入 Web3 或 AI 方向的朋友。如果你已经看过很多资料，但还不知道怎么展示自己的进展，这场活动会比较有帮助。

先讲一个小故事。

有一个新人，我们暂时叫他小林。小林连续两周都在学习 Web3，他问 AI 解释过 NFT、钱包、智能合约、Gas，也跟着教程跑过一个简单页面。他每天都觉得自己学了很多，但到了组队环节，别人问他“你能提供什么 Proof of Work”，他突然卡住了。

因为他没有公开记录，没有 README，没有 Demo 截图，也没有一段能说清楚自己做了什么的文字。更尴尬的是，他明明用了 AI 帮自己学习，却说不清 AI 帮了什么，自己又做了哪些判断。

今天我们带着一个问题来听完整场活动：

```text
如果你是小林，你怎么把这两周的零散学习，整理成一份别人看得懂、愿意相信的 Proof of Work？
```

这个问题先不急着回答。我们会在后面的案例拆解、互动和 Q&A 里一步步回到它。

### 问题引入 8-15 分钟

很多新人不是没有学习，而是学习过程没有留下痕迹。比如问了 AI 很多问题、看了很多项目、试着跑过代码，但最后别人看不到你的理解，也看不到你能承担什么角色。

所以今天我们不只讨论怎么用 AI，而是讨论怎么把 AI 辅助变成自己的学习记录、项目文档和作品证明。

回到小林的故事，他真正缺的不是“再多学一个概念”，而是把已有学习变成结构化输出。也就是回答清楚三件事：我围绕什么问题学习，我具体做出了什么，我下一步准备验证什么。

### 案例拆解 15-30 分钟

接下来我们用一个新人 Builder 的视角来看：如果我正在做一个 Web3 小项目，比如 NFT 粉丝徽章、链上积分、AI Badge 或活动运营方案，我可以让 AI 帮我做什么？

AI 可以帮忙解释概念、拆解任务、整理 README、形成开发计划，也可以帮我们把“我做了什么、我为什么这样做、我下一步要验证什么”写清楚。但这里有一个重点：AI 不能替代真实理解，最终的判断、验证和取舍仍然要由人完成。

如果把这个方法放回小林身上，他可以把“我问了 AI 很多问题”改写成更有证明力的表达：

```text
我围绕 NFT 粉丝徽章这个主题学习了钱包连接、NFT 合约和链上积分。
我整理了一份学习笔记，并用 AI 辅助生成 README 初稿，随后根据自己的项目理解进行了修改。
我下一步会验证 mint 流程是否能在测试网上跑通，并补充截图和交易哈希。
```

### 核心讨论 30-45 分钟

这一部分可以由主持人向嘉宾提问，也可以让参与者在评论区提交问题。讨论重点放在新人最容易卡住的地方：不知道写什么、不知道怎么证明自己、不知道 AI 输出是否可信。

主持人可以在每个问题后都回扣小林案例：如果小林只有聊天记录，够不够？如果小林有 README 但没有验证截图，可信度差在哪里？如果小林不是 Dev，而是 Ops 或 Product，他还能怎么留下 Proof of Work？

### 互动与 Q&A 45-55 分钟

现在请大家用 3 句话写一个自己的 Proof of Work 草稿。它不需要完美，只需要说明：

1. 我正在学习或尝试什么主题。
2. 我已经完成了什么具体输出。
3. 我下一步希望验证或改进什么。

主持人可以邀请 2-3 位参与者分享，并现场帮他们把表达变得更清楚。

随后进入一个短 Q&A，把暖场里的问题拿回来分析：

```text
如果你是小林，你会怎么补救？
```

可以引导大家给出答案：

- 把 AI 问答整理成一篇学习笔记，而不是只保留聊天记录
- 给项目补一个 README，说明主题、目标用户、核心功能和当前进度
- 加入截图、交易哈希、GitHub commit 或活动文档作为证据
- 单独写一段“AI 帮助了什么，我自己判断了什么”
- 用 3 句话说明自己能在团队里承担的角色

这样开头留下的悬念在 Q&A 阶段被解决：Proof of Work 不是等项目完全成功后才出现，而是把学习、验证和输出持续记录下来。

### 收尾 55-60 分钟

今天的重点不是让大家记住所有方法，而是带走一个行动：把你最近的一次学习或实践整理成公开记录。它可以是一篇 Markdown、一份 README、一条长推、一份活动复盘，或者一个 GitHub 文件。

只要你能持续留下清晰的学习轨迹，你就已经在积累 Builder 的 Proof of Work。

## 核心问题

### 问题一：新人为什么需要学习记录？

可以讨论：

- 学习记录和普通笔记有什么区别
- 为什么 Builder 需要让别人看见自己的思考过程
- 没有完整项目时，如何证明自己正在进步

### 问题二：AI 可以帮助新人完成哪些具体输出？

可以讨论：

- 概念解释
- 学习路径整理
- README 初稿
- 项目计划
- 活动复盘
- 角色说明
- GitHub 文档

重点强调：AI 是辅助工具，不是最终责任人。

### 问题三：如何判断 AI 输出是否能放进自己的 Proof of Work？

可以讨论：

- 是否经过自己理解和修改
- 是否有真实项目或学习背景
- 是否能被代码、截图、链接或过程记录验证
- 是否避免把 AI 生成内容直接当成事实

### 问题四：没有技术背景的人也能做 Builder 记录吗？

可以讨论：

- Ops 可以记录活动策划、用户反馈和社区增长
- Product 可以记录需求拆解、用户流程和竞品分析
- Design 可以记录界面方案、视觉探索和用户体验
- Research 可以记录项目调研、概念解释和资料整理

Builder 不只等于写代码，关键是持续产出、验证和协作。

### 问题五：一份好的新人 Proof of Work 应该包含什么？

可以讨论：

- 明确主题
- 具体问题
- 自己做过什么
- AI 帮助了什么
- 自己做了哪些判断和修改
- 当前结果
- 下一步计划

## 互动设计

互动名称：3 句话 Proof of Work 草稿

互动目标：让参与者在活动中立刻完成一个最小输出，而不是只听分享。

参与方式：

```text
第 1 句话：我正在学习或尝试的主题是____。
第 2 句话：我已经完成的具体输出是____。
第 3 句话：我下一步想验证或改进的是____。
```

示例：

```text
我正在学习如何用 AI 辅助设计 Web3 新人活动。
我已经完成了一份活动主题分析和一份活动流程草稿。
我下一步想验证这个主题是否真的能帮助新人产出自己的 Proof of Work。
```

主持人可以选择 2-3 个参与者的回答进行现场优化，帮助他们把表达从“我学了很多”改成“我围绕某个问题完成了某个输出”。

## 暖场案例回收

活动最后可以再次回到小林的故事，帮助参与者理解完整闭环。

开场时，小林的问题是：

```text
我学了很多，但我拿不出 Proof of Work。
```

活动结束时，小林可以把自己的经历整理成：

```text
我正在学习 Web3 NFT 应用，重点理解粉丝 Badge、链上积分和 AI 辅助开发流程。
我已经完成了 NFT 概念学习笔记、一个项目 README 初稿，以及一次 AI 协作记录整理。
我下一步会补充测试网运行截图、交易哈希和 GitHub commit，验证这个原型是否真的可运行。
```

这个前后对比就是本场活动希望让新人带走的核心方法：不要只说“我学了很多”，而是说清楚“我围绕什么问题，完成了什么输出，下一步验证什么”。

## 收尾 CTA

活动结束后，邀请参与者在 24 小时内完成一个小行动：

```text
发布一份自己的 Builder 学习记录。
```

可选形式包括：

- 一篇 Markdown 学习笔记
- 一个 GitHub README
- 一条长推或社区帖子
- 一份活动复盘
- 一个小 Demo 的截图和说明
- 一份 300-500 字的角色说明

建议发布时包含四个部分：

1. 我选择的学习主题
2. 我想解决的问题
3. 我已经完成的输出
4. 我下一步要验证的内容

最后可以用一句话收尾：

```text
不要等到完整项目完成后才开始展示。清晰记录每一次学习和验证，本身就是 Builder 的 Proof of Work。
```
