## 2. 协作方式

AI 没有独立决定项目方向。项目主题、技术栈和功能亮点都来自我的选择，Codex 主要负责把这些想法整理成更清晰、可执行和可提交的方案。

## 3. 近期协作过程

### 3.1 Web3 基础概念学习

我向 Codex 询问了 NFT 合约与 Token 合约的区别，以及 ERC-20 积分与 ERC-721 Badge 如何配合。

Codex 帮助我理解了：

- ERC-20 适合表示可互换的积分。
- ERC-721 适合表示每一枚都可以拥有独立身份和 metadata 的 Badge NFT。
- 粉丝平台可以形成“签到或互动获得积分，再消耗积分兑换 Badge”的流程。
- 链上历史不能被篡改，但智能合约的当前状态可以通过合法交易更新。
- RPC 节点会根据最新区块返回当前合约状态。

我通过继续追问积分增加、链上存储和最新数据读取的方式，核查自己是否真正理解了这些概念，而不是直接复制 AI 的结论。

### 3.2 Solidity 与事件机制学习

我向 Codex 询问了 `_nextTokenId`、`_safeMint`、`emit` 和 `tokenURI` 等 Solidity / ERC-721 概念。

Codex 帮助我理解了：

- `_nextTokenId` 通常可以使用 `uint256`。
- `_safeMint` 会铸造 NFT，并检查合约接收方是否支持 ERC-721。
- `emit` 会把事件写入交易日志，后端可以通过 RPC 查询或订阅。
- `tokenURI` 通常由钱包、NFT 页面或市场在展示 NFT 时主动读取，不一定在铸造函数内部调用。
- Solidity 中声明为 `constant` 的积分规则部署后不能修改；需要动态配置时，应改为状态变量并增加权限控制和更新事件。

这些解释帮助我理解现有合约，但最终的权限设计、积分参数和合约安全性仍需要人工审查和测试。

### 3.3 技术路线比较

我询问了 Solana、Monad、EVM、Layer 1 和 Solidity 的区别。

Codex 帮助我整理出：

- Monad 是 EVM 兼容链，可以继续使用 Solidity 和常见以太坊开发工具。
- Solana 是 Layer 1，主要使用 Rust、Anchor 和 SVM，不原生使用 Solidity。
- 对当前已经使用 Solidity、wagmi 和 EVM 钱包的项目，继续在 Monad 上实现原型成本更低。
- 学习 Solana 的价值主要在于理解不同的账户模型、并行执行方式和 Rust 开发体系。

我最终保留 Monad 作为当前项目链，因为这更符合已有代码和本周交付时间，并没有因为 AI 推荐而临时切换技术栈。

### 3.4 前端与本地运行问题

我询问了 JSX、TS、TSX 文件的区别，以及一个 React 组件的 JSX 语法问题。

Codex 说明了 React 组件应该完整闭合父元素，并建议当前 TypeScript React 项目中的组件使用 `.tsx`，纯逻辑或配置使用 `.ts`。

在本地页面运行方面，Codex 提供了开发模式和静态目录的启动命令。由于运行环境对端口监听有限制，AI 没有把无法启动的服务描述为已经成功，实际页面是否正常仍需要在本机浏览器中验证。

### 3.5 最小 Web3 / AI 原型规划

我提出要定义一个粉丝 NFT 平台的最小功能，并要求说明用户动作、需要阅读的文档、本周真实实现内容、可以 mock 的内容和验收方式。

Codex 将想法整理为以下最小闭环：

1. 用户连接钱包并获得互动积分。
2. 用户达到积分条件后发起 Badge 兑换。
3. 智能合约消耗积分并产生链上记录。
4. NFT 铸造成功后，页面展示 Badge、Token ID 和交易哈希。

我随后补充了两个关键约束：

- 后端使用 Python FastAPI，并结合 Supabase。
- 增加“达到一定积分后，由 AI 动态生成专属 NFT Badge”的产品亮点。

Codex 根据这些要求补充了 FastAPI 接口、Supabase 数据表、Storage、AI 生成任务和链上验证流程。普通 Badge 暂定消耗 50 积分，AI Badge 暂定消耗 100 积分。这些数字目前属于原型参数，正式使用前需要由人类根据运营规则重新确认。

## 5. 人类进行了哪些删改、选择和核查

- 我明确选择了粉丝互动和 NFT Badge 作为项目主题。
- 我决定使用 ERC-20 / 链上积分与 ERC-721 Badge 形成产品闭环。
- 我选择 Monad 和 Solidity，而没有因为技术讨论切换到 Solana。
- 我指定后端使用 Python FastAPI，数据服务使用 Supabase。
- 我要求增加 AI 动态生成 Badge，并要求只有达到积分条件的用户才能使用。
- 我选择 Dev 作为主方向，并将 Week 3 角色定位为 Web3 全栈开发。
- 我通过连续追问核查了 AI 对 Solidity、事件、链上存储和 Token URI 的解释是否符合我的理解。
- 我要求 AI 只提交指定文档，没有授权它提交当前尚未完成的合约和前端修改。

仍需由我进一步核查的内容：

- 50 分和 100 分是否是合理的兑换成本。
- 智能合约是否真正安全地扣除积分，是否存在重复兑换或权限绕过。
- FastAPI 是否正确验证链上交易、钱包地址和事件参数。
- Supabase 的 Row Level Security、Storage 权限和敏感字段是否配置正确。
- AI 生成图片是否符合内容安全、版权和社区规范。
- 测试网部署、完整用户流程和失败状态是否真实可运行。

## 6. 哪些事情不能直接交给 AI

- 不能把钱包私钥、助记词、Supabase Service Role Key 或生产环境密钥交给 AI。
- 不能让 AI 在没有人工确认的情况下操作主网资产、签署交易或执行不可逆部署。
- 不能直接相信 AI 生成的智能合约已经安全，必须进行测试、代码审查，必要时进行专业审计。
- 不能让 AI 独立决定积分经济、用户权益、收费方式和治理规则，这些属于产品和运营责任。
- 不能让 AI 独立判断生成图片的版权、肖像权和合规风险。
- 不能把用户隐私数据、未公开业务数据或敏感身份信息直接提供给 AI。
- 不能用 AI 输出代替真实测试。接口、数据库记录、交易哈希、NFT 所有权和页面状态都必须人工验证。
- 不能把 AI 编写的说明当作最终事实，涉及框架版本、链参数和 API 用法时，需要以官方文档为准。

## 7. 协作反思

这次协作中，AI 最有价值的作用不是代替我做决定，而是帮助我快速理解陌生概念、补全方案遗漏并把想法转换成可执行任务。

人类仍然需要负责确定问题、选择技术路线、控制范围、保护密钥、验证运行结果和承担最终决策。对我来说，比较有效的协作方式是先提出一个明确目标，再不断补充约束，并要求 AI 给出可以通过代码、接口、数据库记录和链上交易验证的结果。
